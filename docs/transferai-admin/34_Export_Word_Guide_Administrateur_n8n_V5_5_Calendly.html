Objet du document
Ce guide decrit la V5.5 de l’assistant Chatwoot + n8n TransferAI. La V5.5 n’ajoute pas de nouvelle brique technique majeure ; elle raffine la logique commerciale pour proposer un lien de demonstration seulement si le lead est suffisamment qualifie et si l’email de suivi est deja connu.

1. Resume executif

La V5.5 conserve tout le socle V5 : memoire conversationnelle, reponse IA, qualification JSON, labels et custom attributes. Elle ajoute une regle de conversion plus stricte autour de la proposition de demo et du lien Calendly.

2. Principe metier

  • ne jamais pousser un lien de rendez-vous trop tot
  • ne jamais proposer Calendly sans email connu
  • utiliser le lien seulement si le besoin est concret ou si la demande de demo est explicite
  • continuer la qualification si le lead reste incomplet

3. Workflow cible

Type Nom recommande Role
Version cible ACTIVE - TransferAI Chatwoot Auto Reply V5.5 (Calendly conditionnel) Version candidate avec logique stricte de proposition de demo.
Rollback ACTIVE - TransferAI Chatwoot Auto Reply V5 (Qualification) Version stable a conserver pendant toute la phase de validation V5.5.

4. Webhook V5.5

https://n8n-pxlk.srv1480638.hstgr.cloud/webhook/chatwoot-inbound-v5-5

5. Architecture

La V5.5 garde les 12 noeuds de la V5 :

  1. Chatwoot Webhook V5.5
  2. Filter Chatwoot event
  3. Fetch conversation history
  4. Build OpenAI reply payload
  5. OpenAI Chat Completions (reply)
  6. Format AI reply
  7. Send reply to Chatwoot
  8. Build qualification payload
  9. OpenAI Qualification JSON
  10. Parse qualification + Compute labels
  11. Update Chatwoot custom_attributes
  12. Apply Chatwoot labels

6. Ce qui change par rapport a la V5

  • path du webhook distinct : chatwoot-inbound-v5-5
  • constante de lien Calendly introduite dans le prompt
  • regles plus strictes dans Build OpenAI reply payload
  • temperature de reponse legerement abaissee pour garder un ton plus cadre

7. Regles de proposition de demo

Rappel central
Le lien Calendly n’est pas un reflexe. C’est une option commerciale qui ne doit apparaitre que si le lead est suffisamment mur et qu’un email de suivi est deja connu.
  • Si sender_email est vide : demander d’abord l’email.
  • Si sender_email est connu et que le besoin est concret : la demo peut etre proposee.
  • Si le lead est encore flou : continuer la qualification, sans lien.
  • Le ton doit rester consultatif, jamais agressif.

8. Valeur a configurer

Le noeud Build OpenAI reply payload contient la constante :

REPLACE_WITH_CALENDLY_URL

Exemple attendu :

https://calendly.com/transferai/demo

9. Cas de test recommandes

Cas 1. Lead incomplet sans email

Bonjour, je dirige une PME et je veux automatiser certains process.

Attendu : le bot pose une ou plusieurs questions utiles et demande l’email avant toute proposition de demo.

Cas 2. Lead complet sans demande explicite de demo

Bonjour, je dirige TransLog SA dans la logistique. Nous voulons automatiser le suivi de flotte.

Attendu : reponse contextualisee, qualification progressive, pas de lien pousse trop tot.

Cas 3. Lead chaud avec demande explicite de demo

Bonjour, je m'appelle Marius. Je dirige TransLog SA, une PME ivoirienne dans la logistique avec 30 employes. Nous voulons automatiser le suivi de flotte et nous voulons une demo cette semaine.

Attendu : reponse contextualisee, lien Calendly possible si l’email est connu, labels coherents comme automatisation, lead-chaud et demo-proposee.

10. Procedure de migration V5 vers V5.5

  1. Importer le workflow V5.5 dans n8n.
  2. Remplacer les placeholders de secrets et le lien Calendly.
  3. Tester en parallele de la V5 avec mock ou conversation dediee.
  4. Retirer tout pin / mock data du webhook.
  5. Activer la V5.5.
  6. Modifier dans Chatwoot l’URL webhook de chatwoot-inbound-v5 vers chatwoot-inbound-v5-5.
  7. Observer les executions 24 a 48h avant archivage de la V5.

11. Rollback

En cas de souci, remettre le webhook Chatwoot sur :

https://n8n-pxlk.srv1480638.hstgr.cloud/webhook/chatwoot-inbound-v5

12. Regles de stabilite

  • ne jamais tester la V5.5 directement sur tous les leads sans observation initiale
  • garder la V5 active comme filet de securite
  • surveiller le comportement sur les leads chauds et les demandes de demo
  • verifier que l’email n’est jamais redemande s’il est deja connu
  • verifier qu’aucun lien Calendly n’est envoye si l’email manque

13. Suite logique

Apres stabilisation V5.5, la suite recommandee reste la meme : V6 pour le handoff humain conditionnel, puis V7 pour la supervision et les logs analytiques.
